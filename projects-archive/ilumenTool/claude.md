# CLAUDE.md — Project Guidelines for IlumenTool

## Project Overview

IlumenTool is a **cross-platform Flutter desktop application** (Linux + Windows) for iLumen BV. It covers the full lifecycle of iLumen IoT hardware products: firmware flashing, EEPROM cloning, automated testing, production batch management, and field installation/commissioning.

**Repository**: `RdFutech/ilumenTool` — branch `iLumenTool_V2.0`  
**Flutter project root**: `ilumentool/` (one level below repo root)  
**Current version**: `2.0.0-alpha.8`  
**Dart SDK constraint**: `>=3.2.0 <4.0.0` (null-safe)

## Architecture & Key Patterns

### Secrets Management

- **`FIREBASE_API_KEY`** is a compile-time constant in `env.dart` (public by design — security via Firestore rules).
- **All other secrets** are fetched at runtime from Firestore `AppConfig/secrets` after Firebase Auth login, via `Env.loadSecrets()`.
- **Never hardcode credentials** in source. Use `Env.someGetter` — all getters throw if called before `loadSecrets()`.
- **Never log credentials** — use `[REDACTED]` or character-count only (e.g., `print("token: [REDACTED] (${token.length} chars)")`).

### Version Management

- `lib/version.dart` contains `const String appVersion` — **auto-generated by `release.sh`**, do not edit manually.
- `pubspec.yaml` version field is also updated by the release script.
- Firebase `ToolInfo/info` document stores `latestVersion`, `minimumVersion`, and `downloadUrl` for auto-updates.

**Product Registry** (`lib/variable_classes/product_registry.dart`): Central single source of truth for all product configuration. Contains:

- `ProductFamily` enum — 9 families (ILB, ILB2, ILC, ILC2, ILH, ILH-SGR, ILS, PBM, SPS) with `.code` and `.fromCode()` factory
- `ProgramTarget` enum — 8 programming targets with per-target config (MCU, ESP, USB, serial, Firebase flags)
- `ProgrammingMode` enum — `parallel` (default) or `sequential` (ILH)
- `ProgramTargetConfig` — per-target configuration: hasEsp, hasAvr, usesCpUsb, needsSerialNumber, firebaseEnabled, hasBootloader, needsEeprom, avrBaudRate, avrProgrammerTool, programmingMode, needsProductInfoInit
- `ProductDefinition` — display name, image file, page title, isSmart flag, sub-types, legacy XML codes
- `FirmwareScanConfig` — pathPrefix + extension for firmware file scanning
- `ProductRegistry` static class — maps for TB device types, TB shared/server attributes, AVR signatures/fuses/parts/bootloader paths/firmware paths, ESP asset paths/firmware paths, firmware scan paths/aliases, build order types, and convenience methods

When adding a new product, update **all** of these locations:

1. `product_registry.dart` — add `ProductFamily` enum value, `ProductDefinition` entry, program target(s), TB mappings, AVR/ESP configs as needed
2. `product_button_widget.dart` — navigation case (display name auto-resolved via registry)
3. `programmingTab.dart` — product button grid
4. `espressif.dart` — SPIFFS config JSON (`burnEspConfig()` — not yet registry-backed)
5. `ProductionView.dart` — dropdown menu item (sub-types auto-resolved via registry)

- Uses `package:flutter_lints/flutter.yaml` (see `analysis_options.yaml`), with ~30 noisy style rules disabled for the legacy codebase.
- `old_files/` directory is excluded from analysis.
- **0 warnings, 0 errors** is the current baseline — maintain it.
- **Current state (March 2026)**: 0 errors, 0 warnings, 4 info-level issues remaining (`unnecessary_overrides` ×2, `collection_methods_unrelated_type` ×1, `unrelated_type_equality_checks` ×1). Down from 46 issues at start of lint cleanup.

### Imports

- Use package imports: `import 'package:ilumentool/...'` (not relative paths).

### Error Handling

- Most API calls currently lack try-catch — **always add error handling** when writing new code.
- Use `print()` (not `debugPrint()`) for logging that must be visible in release builds.

### Process Execution

- `burnEspFlash()` uses `Process.start()` with stream-based output (`stdbuf -o0 -e0` on Linux) for real-time progress tracking. Concurrent stdout/stderr drain via `Future.wait()`. USB port released with `await exitCode`.
- Other esptool operations (bootloader, partitions, boot0app, config, erase) still use `Process.run()` — these are small/quick and don't need progress streaming.
- AVR programming via avrdude uses `Process.run()` — blocking.
- Platform-specific paths: always handle both Linux (`/`) and Windows (`\\`). Prefer the `path` package where possible.

### TODO Management

All task tracking lives in **`docs/TODO.md`** — **never add `// TODO` comments in source code**.

When creating or updating `docs/TODO.md`, follow this exact format:

- **Section headers** match the `lib/` folder structure: `## Views (lib/views/)`, `## Connectors (lib/connectors/)`, etc.
- **Sub-headers** are the file name in backticks: `### \`programmingView.dart\``
- Every section uses the **same 4-column table**:

  ```markdown
  | # | Phase | Status | Priority | Description |
  | --- | --- | --- | --- | --- |
  | 1 | P1 | Open | High | Short, clear description of the task |
  ```

- **Phase** values: `P0` (security, done), `P1` (cleanup & core features), `P2` (architecture), `P3` (view decomposition), `P4` (testing/logging/CI), `PX` (extras — new features/integrations, independent of core phases). Must match the phase definitions in the Roadmap Summary section of this file.

- **Markdown table styling** — use **compact style only** (no column padding):
  - Header separator: `| --- | --- | --- | --- |` (not `|----|--------|----------|-------------|`)
  - Data cells: `| 1 | Open | High | Description |` (not `| 1  | Open   | High     | …`)
  - This prevents MD060 (`table-column-style`) linting warnings caused by misaligned pipes when cell content exceeds header width.
- **Status** values: `Open`, `In Progress`, `Done`.
- **Priority** values: `High`, `Medium`, `Low`.
- **#** column is a unique sequential number across the whole file — never reuse or renumber.
- When closing an item, change Status to `Done` and leave the row (don't delete it).
- **Summary table** at the bottom must be kept up to date with counts.
- Keep a `## Code Quality` section for lint / refactoring tasks.
- Add new items at the end of the relevant file section; use the next available `#`.

### Markdown Hygiene

- A `.markdownlint.json` config lives at the repo root. Disabled rules: `MD013` (line-length — impractical for tables) and `MD060` (table-column-style — conflicts with compact-style convention).
- **Run after every `.md` edit:** `npx markdownlint-cli '**/*.md' --ignore old_files --ignore node_modules`
- Common pitfalls to avoid:
  - **MD031** — Fenced code blocks must have a blank line before and after.
  - **MD032** — Lists must have a blank line before and after (including numbered lists and lists inside blockquotes).
  - **MD040** — Fenced code blocks must specify a language (use `text` for plain text, `bash` for shell, `dart` for Dart).
  - Inside blockquotes, a list needs a blank `>` line before it:

    ```markdown
    > Some text:
    >
    > - List item 1
    ```

### Analyzer Hygiene

- **Target: 0 warnings, 0 errors** at all times. Run `flutter analyze` before committing.
- ~30 noisy style lint rules are currently disabled in `analysis_options.yaml` (see `docs/TODO.md` #39 for the plan to re-enable them incrementally).
- When removing a field or variable, always grep for **all references** (assignments, reads, comments) and clean them up in the same change — do not leave dangling references.
- For unused code that is intentionally kept for future use, add `// ignore: unused_element` (or `unused_field`, `unused_local_variable`) on the line **above** the declaration.
- Never suppress warnings on code that is truly dead — delete it instead.

## Active Migrations & Known Issues

### Build Orders (implemented March 6, 2026)

Full build order management system for production planning. Uses Firestore collection `build_orders/{orderId}`.

**Firestore document schema** (`build_orders` collection):

```text
build_orders/{orderNumber}
├── orderNumber: string          ← "BO-YYYYMMDD-XXXX" (auto-generated)
├── status: string               ← "draft" | "approved" | "in_progress" | "completed" | "cancelled"
├── customerName: string?
├── customerReference: string?   ← Customer PO number
├── description: string?
├── priority: string?            ← "low" | "medium" | "high"
├── requestedBy: string?
├── approvedBy: string?
├── createdBy: string?
├── updatedBy: string?
├── createdAt: string (ISO 8601)
├── updatedAt: string (ISO 8601)
├── dueDate: string? (ISO 8601)
├── completedAt: string? (ISO 8601)
├── notes: string?
├── assignedSerials: string[]    ← Serial numbers linked to this order
└── items: array                 ← Line items
    └── [i]
        ├── productType: string  ← "ILB2" | "ILC" | "ILC2" | "ILH" | "ILH-SGR" | "ILS" | "PBM" | "SPS"
        ├── subType: string?     ← e.g. "ILB2.5", "ILB5.0", "SPS15"
        ├── quantity: int
        ├── completedQuantity: int
        ├── firmwareVersion: string?
        └── notes: string?
```

**Status workflow**: `draft` → `approved` → `in_progress` → `completed` (or `cancelled` at any non-complete stage).

**Files**:

- `lib/variable_classes/build_order.dart` — `BuildOrder`, `BuildOrderItem`, `BuildOrderStatus` data classes
- `lib/services/build_order_service.dart` — Firestore CRUD (create, read, list, update, status transitions, serial assignment, soft-delete)
- `lib/services/build_order_pdf_service.dart` — A4 PDF export (header, order info, line items table, serials, notes, signature block)
- `lib/views/build_order_list_view.dart` — List with status filter dropdown, stat cards, progress bars
- `lib/views/build_order_form_view.dart` — Create/edit form with multi-line-item support
- `lib/views/build_order_detail_view.dart` — Detail view with status actions (approve/start/complete/cancel), PDF export button

**Tab access**: Users need `BuildOrders: true` in their Firestore `Users/{email}/tabs/tabs` document (same pattern as other tabs). Can be toggled by an admin via the Gebruikers (Users) tab — click on a user, then tap the "BuildOrders" checkbox.

**Firestore security rules**: The `build_orders` collection requires a rule in Firebase Console:

```text
match /build_orders/{orderId} {
  allow read, write: if request.auth != null;
}
```

This is documented in `firestore.rules` (reference file, deployed manually via Firebase Console).

### Firebase Platform Abstraction Layer (implemented)

- **Official Firebase SDKs (`firebase_core`, `firebase_auth`, `cloud_firestore`) do NOT support Linux** at runtime — they rely on native platform channels that only exist for Android, iOS, macOS, Web, and Windows.
- Attempted full migration: replaced `firedart` with official SDKs in all 8 files. Build succeeded but app crashed at launch with `PlatformException(channel-error, Unable to establish connection on channel)`.
- **`firebase_dart`** (pure Dart Firebase) was evaluated but only supports Core, Auth, Database, Storage — **NOT Cloud Firestore**.
- **Solution implemented**: Platform abstraction layer in `lib/services/firebase/`:
  - `firebase_interface.dart` — abstract `FirebaseBackend` interface
  - `firedart_backend.dart` — firedart implementation (works on Linux + Windows)
  - `official_sdk_backend.dart` — official SDK implementation (Windows only, stub for now)
  - `firebase_factory.dart` — factory that selects backend based on `Platform.isLinux`
  - `firestore_device_service.dart` — high-level device CRUD using the abstraction
- App code currently uses `firedart` backend on all platforms. Official SDK backend is ready to be wired in when Windows-only builds are needed.
- **`ilumentool_db`** added as git dependency — uses official SDKs internally. Available as a transitive dependency but **not wired into app logic** until Linux Firestore support is resolved (Phase 2).
- Dart SDK constraint bumped to `>=3.2.0 <4.0.0` (compatible with both `firedart ^0.9.8` and `ilumentool_db`).
